# Phase 7 : Internationalisation (FR/EN)

## Résumé

L'interface du **Git Visualizer** est actuellement entièrement en **français**. Cette spec ajoute le support de l'**anglais (EN)** via un système d'internationalisation réactif : sélecteur de langue en UI, dictionnaires de traduction externalisés, persistance du choix, et **frontière stricte** avec le moteur Git (les messages d'erreur de Git restent en anglais, fidèles au vrai `git`).

**Décision clé** : Les messages d'erreur du core (retournés par `CommandResult.fail()`) conservent l'anglais (calqués sur Git) ; les libellés d'interface (sidebar, modales, placeholders, `git help`, autocomplétion, scénarios) sont traduits. Cela préserve le déterminisme (pas de langue -> pas d'affectation des hashes/commandes).

## Scope

**Textes à traduire (FR ↔ EN)** :
- Libellés UI : "Branches", "HEAD", "Tags", "Opération en cours", etc. (RefsSidebar)
- Placeholder : "Graphe Git", "Initialisez un dépôt...", "Aucun commit..." (GraphView)
- Terminal & panneau : "Commandes récentes", "Reset History", prompts, hints
- Modale interactive rebase : labels de actions ("pick", "reword", "squash", "fixup", "drop")
- `git help` : descriptions des commandes (du catalogue + synopsis/flags)
- Scénarios : titre, description, objectifs
- Tutoriels pédagogiques : si ajoutés en Phase 7+
- Boutons & actions : [Continue], [Abort], [Execute Scenario], etc.

**Textes hors scope** :
- Messages d'erreur Git (core) : restent EN
- Noms de commandes/flags : restent immuables (`commit`, `-m`, `--soft`, etc.)
- Hashes et contenu des commits : inchangés
- Code/log d'exécution dans le terminal : non traduits

### 4. Composable i18n

**`src/i18n/index.ts`** :
```typescript
import { ref, computed } from 'vue';
import type { MessageKey } from './messages';
import frMessages from './locales/fr.json';
import enMessages from './locales/en.json';
import { getCommandCatalog as getCommandCatalogFr } from './catalogs/fr';
import { getCommandCatalog as getCommandCatalogEn } from './catalogs/en';

export type Locale = 'fr' | 'en';

const locales: Record<Locale, Record<string, string>> = {
  fr: frMessages,
  en: enMessages,
};

const catalogsByLocale: Record<Locale, CommandCatalog> = {
  fr: getCommandCatalogFr(),
  en: getCommandCatalogEn(),
};

// État réactif
const currentLocale = ref<Locale>('fr');

// Composable
export function useI18n() {
  const t = (key: MessageKey): string => {
    return locales[currentLocale.value][key] ?? locales['en'][key] ?? key;
  };

  const setLocale = (locale: Locale) => {
    currentLocale.value = locale;
    // Persister dans localStorage
    localStorage.setItem('i18n.locale', locale);
  };

  const getLocale = (): Locale => currentLocale.value;

  const getCatalog = (): CommandCatalog => catalogsByLocale[currentLocale.value];

  return { t, setLocale, getLocale, getCatalog };
}

// Initialisation au démarrage
export function initI18n() {
  const saved = localStorage.getItem('i18n.locale');
  if (saved === 'en' || saved === 'fr') {
    currentLocale.value = saved;
  } else {
    // Déduire de `navigator.language` (optionnel)
    const lang = navigator.language?.startsWith('en') ? 'en' : 'fr';
    currentLocale.value = lang;
  }
}
```

### 5. Catalogue de commandes traduit

**Décision** : Le catalogue `COMMAND_CATALOG` (actuellement monolithique dans `src/core/catalog.ts`) doit exister en deux versions : FR et EN. Les descriptions (synopsis, flags, exemples) varient ; les noms de commandes (`add`, `commit`, etc.) restent identiques.

**Approche** :
- **Option A (propre)** : Externaliser le catalogue en données (JSON) + traductions séparées.
- **Option B (pragmatique)** : Dupliquer le catalogue dans `src/i18n/catalogs/fr.ts` et `en.ts`, généré depuis un modèle partagé.

### 8. Persistance & synchronisation

**localStorage** :
```javascript
// Clé unique
const LOCALE_STORAGE_KEY = 'i18n.locale';

// Sauvegarder au changement
useI18n().setLocale('en'); // Appel dans LanguageSwitcher.vue
// → localStorage.setItem('i18n.locale', 'en')

// Charger au démarrage de l'app
initI18n(); // Appelé dans main.ts ou App.vue setup
// → Lit localStorage.getItem('i18n.locale')
```

**Synchronisation multi-onglet** (optionnel Phase 7, optionnel Phase 8) :
```typescript
// Écouter les changements de localStorage depuis un autre onglet
window.addEventListener('storage', (e) => {
  if (e.key === LOCALE_STORAGE_KEY && e.newValue) {
    currentLocale.value = e.newValue as Locale;
  }
});
```

### 9. Intégration du moteur (core)

**Pas de changement au moteur** :
- `GitEngine` et ses commandes restent **zéro-localisés** (code, messages d'erreur en EN).
- Le core exporte `getCommandNames()`, `getCommandFlags()`, etc., qui consultent le catalogue du core (EN).
- L'UI appelle `i18n.getCatalog()` pour obtenir la version traduite et l'affiche (RefsSidebar, `git help`, autocomplétion).

**Contrat** :
- `engine.getCatalog()` → catalogue EN (du core)
- `i18n.getCatalog()` → catalogue traduit (FR ou EN selon la locale)

### 10. Snapshot & déterminisme

**Règle inviolable** : La locale n'affecte **jamais** les résultats des commandes, les hashes ou l'état du moteur.

- Messages d'erreur retournés par `CommandResult.fail()` : EN uniquement
- Snapshot de l'UI (branche, tags, commits) : zéro impact locale
- Hashes déterministes : inchangés en toutes langues

## Cas d'utilisation

### CU-i18n-01 : Affichage initial en FR (par défaut)

L'utilisateur arrive pour la première fois.

```
localStorage ne contient pas 'i18n.locale'
→ initI18n() choisit 'fr' (défaut)
→ L'UI affiche en français
→ localStorage.setItem('i18n.locale', 'fr')
```

### CU-i18n-02 : Basculer en EN

L'utilisateur clique sur le sélecteur de langue et choisit "English".

```
LanguageSwitcher.vue → setLocale('en')
→ localStorage.setItem('i18n.locale', 'en')
→ Tous les composants utilisant t() se rafraîchissent (ref réactif)
→ UI entièrement en EN
→ Aide (`git help`) en EN
→ Autocomplétion proposant des déscriptions EN
```

### CU-i18n-03 : Recharger la page en EN

L'utilisateur revient après avoir changé en EN.

```
initI18n() lit localStorage.getItem('i18n.locale') → 'en'
→ currentLocale.value = 'en'
→ L'UI s'affiche directement en EN
```

## Critères d'acceptation

### CA-i18n-01 : Messages type-safe

**Given**
- `messages.ts` définit une union `MessageKey` de toutes les clés

**When**
- Un développeur appelle `t('invalid.key')` en TypeScript

**Then**
- Erreur de typage à la compilation (TS strict)
- IDE autocomplete suggère les clés valides

### CA-i18n-02 : Fichiers JSON bien formés

**Given**
- `locales/fr.json` et `locales/en.json` existent

**When**
- Le build Vite s'exécute

**Then**
- Les JSON sont valides (pas d'erreur parse)
- Tous les fichiers contiennent les mêmes clés (vérifier en test)

### CA-i18n-03 : Composant reflex sur changement de locale

**Given**
- `LanguageSwitcher.vue` montée
- `RefsSidebar.vue` montée avec `{{ t('sidebar.branches') }}`

**When**
- Utilisateur change de langue via dropdown

**Then**
- `currentLocale.value` change
- RefsSidebar se rafraîchit automatiquement
- Texte passe de "Branches" à "Branches" (idem) ou "Commandes récentes" → "Recent commands"
- Aucun rechargement page

### CA-i18n-04 : Persistance locale

**Given**
- Utilisateur a choisi "English"

**When**
- Recharge la page (F5)

**Then**
- `initI18n()` lit localStorage
- L'UI affiche en EN sans rechargement supplémentaire
- localStorage contient `'i18n.locale': 'en'`

### CA-i18n-05 : Messages d'erreur Git restent EN

**Given**
- Utilisateur en locale FR
- Exécute `git commit` sans rien staged

**When**
- La commande échoue

**Then**
- Message retourné par le core : EN (ex. "nothing added to commit")
- Affiché dans le terminal : EN
- PAS traduit par l'i18n (respect de la fidélité git)

### CA-i18n-06 : Catalogue traduit en `git help`

**Given**
- Utilisateur en locale FR
- Exécute `git help commit`

**When**
- Moteur exécute la commande

**Then**
- Description (depuis `i18n.getCatalog()`) : en français
- Synopsis, flags, exemples : en français
- Si utilisateur bascule en EN et re-exécute `git help commit` → EN

**Automatisation** : Tester que `git help <cmd>` consomme bien le catalogue traduit

### CA-i18n-07 : Autocomplétion traduite

**Given**
- Utilisateur en locale EN
- Tape `git ` puis Tab

**When**
- Appel `autocomplete(...)`

**Then**
- Candidats listés : noms de commandes (invariants)
- Descriptions dans la liste (optionnel) : EN

### CA-i18n-08 : Scénarios traduits

**Given**
- `scenarios.ts` définit un scénario avec titre/description en FR

**When**
- Utilisateur bascule en EN

**Then**
- Titre et description du scénario affichent la version EN (ou fallback à FR si non traduit)
- Cas d'usage : `ExecuteScenarioModal` affiche les détails en langue courante

**Automatisation** : Vérifier que les titres/descriptions de scénarios consultent la locale

### CA-i18n-09 : Default fallback EN

**Given**
- Une clé manque dans `locales/fr.json` (bug)
- Utilisateur en FR

**When**
- Composant appelle `t('missing.key')`

**Then**
- Fallback à EN : `locales/en.json['missing.key']`
- Si absent aussi → retourner la clé elle-même (ex. `'missing.key'`)
- Pas d'exception

### CA-i18n-10 : Sélecteur de langue accessible

**Given**
- `LanguageSwitcher.vue` rendu

**When**
- Navigation clavier : Tab → select focus
- Arrow down → changer option
- Enter → appliquer

**Then**
- UI mise à jour
- Pas de bug focus/ARIA

**Automatisation** : Tester le markup HTML (label associé, options correctes)

### CA-i18n-11 : Pas d'impact sur les hashes

**Given**
- User exécute `git init`, `git add f.txt`, `git commit -m "test"` en FR

**When**
- Bascule en EN, re-exécute les mêmes commandes dans un nouveau dépôt

**Then**
- Hashes des commits identiques
- Snapshot identique (sauf locale UI)
- Comportement Git invariant

**Automatisation** : Comparer hashes entre deux exécutions FR et EN

### CA-i18n-12 : Catalogue bien synchronisé

**Given**
- Une nouvelle commande ajoutée au core (ex. `git foo`)

**When**
- Catalogues FR et EN doivent être mis à jour simultanément

**Then**
- Les deux contiennent la commande
- Pas d'asymétrie (ex. commande absente d'une version)
- Tests vérifient la parité des clés

**Automatisation** : Vérifier que `Object.keys(catalogFr.commands) === Object.keys(catalogEn.commands)`

## Implémentation : Points clés

1. **Composable `useI18n()`** : réactif, exporté dans tous les composants nécessaires

2. **Type-safety** : Union `MessageKey` pour validation à la compilation

3. **LocalStorage** : clé `'i18n.locale'`, valeurs `'fr'` ou `'en'`

4. **Catalogues dupliqués** (Phase 7) : deux fichiers `catalogs/fr.ts` et `en.ts` avec même structure, traduits

5. **Pas de logique git** : Aucune mutation du moteur selon la locale

6. **Fallback EN** : Toute clé manquante en FR → chercher en EN

7. **Réactivité** : Changement de locale → tous les composants utilisant `t()` se rafraîchissent

## Dépendances inter-phases

- **Phase 7** : Implémentation de base (composable, fichiers JSON, intégration RefsSidebar/GraphView/TerminalPanel).
- **Phase 8** : Éventuel refactor de `catalog.ts` vers une structure unifiée + traductions éternalisées.
- **Phase 9** : Internationalisation des **tutoriels** et **éditeur de conflits** (si ajoutés).

## Notes pour Phase 7+

- **Maintenance** : Ajouter nouvelles clés systématiquement dans les deux fichiers JSON.
- **Tests** : Vérifier la parité FR/EN (test qu'aucune clé ne manque).
- **Perf** : Catalogues en TS (pas réimportés à chaque fois), compilés en JS au build.
- **Optionnel Phase 8** : Ajouter support de langues supplémentaires (ES, DE, etc.).
